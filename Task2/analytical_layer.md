# Task 2 — Аналитический слой с учётом Privacy by Design

## 1. Концепция аналитического слоя

Аналитический слой строится по принципу **многоуровневого хранилища (Medallion Architecture)**
с встроенной классификацией и защитой данных на каждом переходе.

Основная задача: позволить BI/ML/AI работать с данными компании, не нарушая конфиденциальность.

---

## 2. Архитектура слоёв хранилища

### Слой 0: Landing Zone (Зона приземления)
```
Источники данных:
  - PostgreSQL (оперативные сервисы)
  - 1С (ERP/бухгалтерия) — выгрузка через адаптер
  - Excel-файлы (исторические данные, миграция)
  - Lab Integration Service (результаты анализов)

Характеристики:
  - Сырые данные в исходном формате (JSON, CSV, Parquet)
  - Данные НЕ изменяются и НЕ используются напрямую для аналитики
  - Полные оригинальные данные, включая PII и MEDICAL_SECRET
  - Доступ: только Data Classification Engine + Data Engineers (роль DATA_ENGINEER)
  - Хранение: S3-совместимое объектное хранилище (MinIO self-hosted)
  - Шифрование: AES-256 at-rest, TLS in-transit
  - Срок хранения: 90 дней (после — архивируется или удаляется по политике)
```

### Слой 1: Bronze Layer (Бронзовый — Очищенные данные)
```
Процессы:
  1. Data Classification Engine классифицирует каждое поле по тегам:
     PII, MEDICAL_SECRET, FINANCIAL, INTERNAL, PUBLIC
  2. Валидация и очистка данных (дедупликация, форматирование)
  3. Добавление метаданных: источник, дата загрузки, тег чувствительности

Характеристики:
  - Данные структурированы, очищены от технических артефактов
  - PII-поля псевдонимизированы: ФИО → patient_uuid
  - MEDICAL_SECRET поля зашифрованы колоночным шифрованием (pgcrypto / ClickHouse encryption)
  - Маппинг patient_uuid → реальные ПДн хранится ТОЛЬКО в оперативной БД
  - Доступ: Data Engineers, строгий RBAC
  - Формат: Apache Parquet + ClickHouse таблицы
```

### Слой 2: Silver Layer (Серебряный — Обработанные данные)
```
Процессы:
  1. Трансформация и обогащение данных (join-ы, вычисляемые поля)
  2. Агрегации первого уровня (по дню, по специалисту — без ФИО)
  3. Обезличивание: удаление псевдонимов, замена на статистические профили
  4. Data Quality checks: проверка полноты, корректности

Характеристики:
  - Нет прямой идентификации пациента (ни ФИО, ни patient_uuid не передаётся дальше)
  - Используются только синтетические или агрегированные идентификаторы
  - Безопасно для аналитиков первого уровня (BI-аналитики)
  - Доступ: BI_ANALYST (роль), Data Scientists (read-only)
  - Формат: ClickHouse таблицы
```

### Слой 3: Gold Layer (Золотой — Бизнес-данные для BI)
```
Процессы:
  1. Финальные агрегаты для бизнес-отчётности
  2. Data Marts (витрины данных) под конкретные отчёты
  3. Автоматическая проверка на отсутствие PII перед публикацией

Примеры витрин:
  - Загруженность врачей (по специализации, без имён пациентов)
  - Выручка по услугам (суммы, без данных плательщика)
  - Популярность услуг по времени года
  - ABC-анализ по категориям услуг
  - Остатки и оборачиваемость склада

Характеристики:
  - Полностью обезличенные агрегированные данные
  - Безопасно для всех аналитиков и менеджмента
  - Доступ: ANALYST, MANAGER, CEO — через BI-инструмент (Superset / Metabase)
  - Формат: ClickHouse MV (Materialized Views)
```

### Слой 4: ML/AI Sandbox
```
Процессы:
  1. Генерация синтетических данных на основе Silver Layer (CTGAN, SDV)
  2. Дифференциальная приватность (Differential Privacy) при обучении моделей
  3. Изолированная среда: нет доступа к Production данным

Характеристики:
  - Только синтетические данные или данные с k-anonymity (k≥5)
  - Изолированный Kubernetes namespace
  - Доступ: DATA_SCIENTIST (через Jupyter с контролем экспорта)
  - Модели обучаются внутри sandbox, результаты (не данные) экспортируются наружу
```

## 3. Поток данных в аналитическом слое

> Визуальная диаграмма потока данных через Medallion Architecture представлена в файле [analytical_layer.drawio](analytical_layer.drawio).

Краткое описание пайплайна:

1. **Источники** (PostgreSQL, 1С, Excel, Lab API) → CDC (Debezium) → Apache Kafka (TLS)
2. **Kafka** → Data Classification Engine (классификация + тегирование полей)
3. **Classification Engine** → Landing Zone (MinIO, AES-256) — сырые данные, 90 дней
4. **Landing** → Apache Spark Job (batch) → Bronze Layer (ClickHouse) — псевдонимизация ФИО → UUID, шифрование MEDICAL_SECRET
5. **Bronze** → Spark Transformation → Silver Layer (ClickHouse) — обезличивание, агрегации без PII
6. **Silver** → ClickHouse Materialized Views → Gold Layer — финальные агрегаты для BI
7. **Gold** → BI Tools (Superset / Metabase) — RBAC: BI_ANALYST видит только Gold
8. **Silver** → ML/AI Sandbox (параллельно) — синтетические данные, differential privacy

## 4. Меры Privacy by Design в аналитическом слое

| Принцип PbD | Реализация в аналитическом слое |
|---|---|
| Минимизация данных | В Gold Layer нет ПДн, только агрегаты |
| Псевдонимизация | Bronze: ФИО → patient_uuid, маппинг только в оперативной БД |
| Обезличивание | Silver: убираем UUID, только статистические профили |
| Шифрование | Landing: AES-256, Bronze: колоночное шифрование MEDICAL_SECRET |
| Контроль доступа | Каждый слой — отдельная роль в Keycloak + ClickHouse RBAC |
| Аудит | Все обращения к Bronze и выше логируются в Audit Log Service |
| Data Lineage | Apache Atlas отслеживает путь данных от источника до витрины |
| Право на удаление | При поступлении запроса: удалить из оперативной БД + очередь на удаление из Landing/Bronze |
| Согласие | Данные пациента попадают в аналитику только при наличии согласия (флаг в CRM) |

## 5. Контроль качества и обнаружение нарушений

### Автоматические проверки при публикации в Gold Layer:
- Сканирование на паттерны ПДн (email, телефон, ФИО) — блокировка публикации при обнаружении
- Проверка k-anonymity: в выборке не должно быть группы менее 5 человек
- Валидация тегов: данные с тегом `PII` не должны попасть в Gold

### Мониторинг и алертинг:
- Алерт: аналитик делает SELECT на Bronze без необходимых разрешений
- Алерт: объём выгрузки из Silver > порогового значения (риск утечки)
- Алерт: новый запрос к колонке с тегом `MEDICAL_SECRET`
