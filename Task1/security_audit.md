## 3. Проблемные зоны (подробный список)

### Зона 1 — Хранение данных в открытом виде
- Excel-файлы с медицинскими картами, журналами пациентов и платёжными данными хранятся на общем диске без шифрования.
- Любой аутентифицированный пользователь домена имеет доступ к файлам.
- Нет разделения по уровням конфиденциальности: медицинская тайна, PII, финансовые данные лежат рядом.

### Зона 2 — Отсутствие аудита доступа
- Нет журнала: кто, когда, с какого устройства открыл, изменил или скопировал файл.
- Невозможно расследовать инцидент утечки данных.
- Нетипичное поведение пользователей (массовое скачивание, доступ в нерабочее время) не отслеживается.

### Зона 3 — Небезопасная передача данных
- E-mail через Exchange без шифрования содержимого (нет S/MIME или PGP).
- Результаты анализов из лаборатории приходят по незащищённому e-mail в виде PDF.
- Направления к врачу содержат диагноз и передаются без защиты.
- ККМ взаимодействует с 1С через TCP/IP + OLE без шифрования на прикладном уровне.

### Зона 4 — Файловый режим 1С
- Базы данных 1С:Бухгалтерия и 1С:Торговля и склад хранятся в виде файлов на диске.
- Нет встроенной СУБД с разграничением доступа на уровне записей.
- Файлы доступны администратору сети без ограничений.
- Нет шифрования на уровне файловой системы.

### Зона 5 — Одна точка отказа
- Вся инфраструктура на одном физическом сервере.
- Нет резервного сервера, нет репликации, нет DR-плана.
- Потеря сервера (пожар, поломка, кража) = потеря всех данных компании.

### Зона 6 — Отсутствие управления доступом на уровне приложений
- В 1С нет настроенных ролей (или они минимальны): бухгалтер видит все данные, кладовщик может открыть кадровые данные.
- В Excel нет разграничения доступа к ячейкам / листам / файлам по ролям.
- Нет принципа минимальных привилегий (least privilege).

### Зона 7 — Двойной ручной учёт
- Кассиры ведут учёт одновременно в Excel и 1С. Это риск расхождений и создание дополнительных незащищённых копий данных.
- Excel-копии финансовых данных могут оказаться на личных устройствах сотрудников.

### Зона 8 — Нет классификации данных
- Отсутствует реестр данных с указанием категории чувствительности.
- Невозможно применить дифференцированные меры защиты.
- Нет понимания, где в системе хранятся данные, относящиеся к медицинской тайне.

### Зона 9 — Работа с лабораторией без интеграции
- Результаты анализов поступают по электронной почте или на бумаге — нет API.
- Ручной процесс привязки результатов к карте пациента создаёт риск ошибок и утечки.
- Нет верификации подлинности данных от лаборатории.

### Зона 10 — Отсутствие обучения сотрудников
- Нет упоминания о политиках информационной безопасности для сотрудников.
- Риск социальной инженерии, фишинга, случайной отправки данных не тому адресату.

## 4. Сравнение с архитектурными практиками Privacy by Design

| Принцип PbD | Статус в AS-IS |
|---|---|
| 1. Проактивность (предотвращение, а не реагирование) | Не реализован |
| 2. Приватность по умолчанию (Privacy as Default) | Не реализован: данные открыты по умолчанию |
| 3. Встроенная приватность (Privacy Embedded) | Не реализован: защита не заложена в процессы |
| 4. Полная функциональность (не в ущерб бизнесу) | Частично: бизнес работает, но без защиты |
| 5. Сквозная безопасность (End-to-End Security) | Не реализована |
| 6. Видимость и прозрачность | Не реализована: нет аудита |
| 7. Уважение к пользователю (согласие, минимизация) | Не реализован |
