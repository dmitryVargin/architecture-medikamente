# Task 1 — Механизм тегирования и защиты данных

## 1. Классификация данных и присвоение тегов

| Тег | Уровень | Описание | Примеры данных |
|---|---|---|---|
| `PUBLIC` | Публичный | Данные без ограничений | Перечень услуг, прайс-лист, адрес клиники |
| `INTERNAL` | Внутренний | Служебная информация | Расписание врачей, остатки склада |
| `CONFIDENTIAL` | Конфиденциальный | Коммерческая тайна | Зарплаты сотрудников, условия договоров с поставщиками |
| `PII` | Персональные данные | ФЗ-152 категория 3 | ФИО, дата рождения, телефон, e-mail, адрес |
| `PII_SPECIAL` | ПДн особой категории | ФЗ-152 категория 1–2 | Диагноз, анамнез, результаты анализов, национальность |
| `FINANCIAL` | Финансовые данные | Платёжная информация | Данные карты, сумма операции, расчётный счёт |
| `MEDICAL_SECRET` | Медицинская тайна | 323-ФЗ ст. 13 | Факт обращения, диагноз, результаты лечения |
| `HR_SENSITIVE` | Кадровые данные | Трудовые отношения | СНИЛС, ИНН, трудовой договор, больничные листы |

## 2. Таблица данных и способы их защиты

| Данные | Тег | В чём необходимость защиты | Способ защиты |
|---|---|---|---|
| ФИО пациента | `PII` | 152-ФЗ: обязательная защита ПДн | Шифрование, псевдонимизация в аналитических слоях |
| Дата рождения | `PII` | 152-ФЗ, в связке с другими данными позволяет идентифицировать | Шифрование |
| Телефон, e-mail | `PII` | 152-ФЗ, риск SPAM / социальной инженерии | Шифрование, маскирование при отображении |
| Адрес (прописка/работа) | `PII` | 152-ФЗ категория 3 | Шифрование |
| Диагноз, анамнез | `MEDICAL_SECRET`, `PII_SPECIAL` | 152-ФЗ категория 1, 323-ФЗ | Шифрование на уровне поля, строгий RBAC |
| Результаты анализов | `MEDICAL_SECRET`, `PII_SPECIAL` | 152-ФЗ, 323-ФЗ | Шифрование, mTLS при передаче, аудит доступа |
| Данные платёжной карты | `FINANCIAL` | PCI DSS, финансовый риск | Токенизация, не хранить PAN в открытом виде |
| Сумма оплаты + услуга | `FINANCIAL`, `MEDICAL_SECRET` | Раскрывает факт обращения и вид лечения | Шифрование, разделение доступа |
| СНИЛС, ИНН сотрудника | `HR_SENSITIVE`, `PII` | 152-ФЗ, налоговая тайна | Шифрование at-rest |
| Зарплата сотрудника | `HR_SENSITIVE`, `CONFIDENTIAL` | Коммерческая тайна | Шифрование, ролевой доступ только HR/бухгалтерия |
| Данные договора с пациентом | `PII`, `FINANCIAL` | Содержит ПДн и финансовые условия | Шифрование, хранение в защищённом хранилище |
| Закупочные цены поставщиков | `CONFIDENTIAL` | Коммерческая тайна | Ограниченный доступ, шифрование |
| Данные клиентского контракта | `PII`, `FINANCIAL` | Содержит паспортные данные, условия, стоимость услуг | Шифрование, электронная подпись, хранение в защищённой схеме |
| Согласие на обработку ПДн | `PII` | 152-ФЗ: юридическая основа обработки | Хранение в CRM, аудит изменений, невозможность удаления без процедуры |
| Расписание врачей | `INTERNAL` | Служебная информация | Ограниченный доступ внутри домена |

## 3. Инструменты тегирования данных

### 3.1 Apache Atlas
- Корпоративный каталог данных с поддержкой классификации (Classification Labels)
- Позволяет создавать иерархии тегов (например, `PII` → `PII_SPECIAL`)
- Поддерживает Data Lineage — отслеживание происхождения и движения данных
- Интегрируется с Hadoop, Kafka, PostgreSQL через плагины
- Применение: центральный реестр всех активов данных с привязкой тегов

### 3.2 OpenMetadata
- Open-source альтернатива Atlas с удобным UI
- Поддерживает теги, glossary terms, классификацию колонок в БД
- REST API для автоматической разметки при загрузке данных
- Применение: каталог данных + UI для ручной разметки аналитиком

### 3.3 Теги на уровне PostgreSQL (будущая СУБД)
```sql
COMMENT ON COLUMN patients.diagnosis IS 'sensitivity:MEDICAL_SECRET;pii:PII_SPECIAL';
```
- Хранение тегов в метаданных таблиц/колонок
- Используется политикой доступа (Row Security Policies)

### 3.4 Теги в Kubernetes / Helm
```yaml
metadata:
  labels:
    data-sensitivity: "medical-secret"
    pii: "true"
```
- Теги на уровне namespace / deployment для сервисов, обрабатывающих чувствительные данные
- Используется в политиках NetworkPolicy для изоляции

### 3.5 Автоматическая классификация (Data Classification Engine)
- Правила на основе регулярных выражений (e-mail, телефон, ИНН, СНИЛС)
- ML-модель для обнаружения чувствительных данных в неструктурированном тексте
- Применяется при загрузке в аналитическое хранилище (см. Task 6)

## 4. Меры защиты на каждом этапе потока данных

### Этап: Ввод данных (пользователь → система)
- TLS 1.3 для HTTPS
- Валидация и минимизация полей: собирать только необходимые данные
- Согласие пациента (consent management) — юридическая основа обработки

### Этап: Обработка (внутри сервиса)
- Данные `MEDICAL_SECRET` и `PII_SPECIAL` — только в памяти, не в логах
- Маскирование в логах (log masking): телефон `+7 900 ***-**-45`
- Принцип минимальных привилегий: сервис получает только нужные поля

### Этап: Хранение (база данных / файловое хранилище)
- Шифрование at-rest: PostgreSQL pgcrypto / TDE, LUKS для дисков
- HashiCorp Vault для управления ключами шифрования
- Раздельное хранение идентификаторов и медицинских данных (псевдонимизация)

### Этап: Передача между сервисами
- mTLS для межсервисного взаимодействия
- API Gateway с проверкой JWT-токена и scope
- Шифрование сообщений в очереди (Kafka — TLS + шифрование сообщений)

### Этап: Интеграция с лабораторией
- API с OAuth 2.0 / mTLS
- Данные передаются в минимальном объёме (только ID пациента, без ФИО)
- Webhook с подписью HMAC для подтверждения подлинности

### Этап: Аналитика / BI
- Обезличивание (anonymization) перед загрузкой в аналитическое хранилище
- Псевдонимизация: замена ФИО на внутренний UUID
- Агрегация: в BI-отчётах нет персональных данных, только статистика

### Этап: Удаление данных
- Политика retention: автоматическое удаление после истечения срока хранения
- Право на удаление (152-ФЗ, «право на забвение»): soft delete → hard delete через 30 дней
- Криптографическое уничтожение (key deletion) для зашифрованных архивов

## 5. Доработанные DFD с мерами защиты

> Визуальные DFD-диаграммы с мерами защиты для каждого процесса представлены в файле [dfd_protected.drawio](dfd_protected.drawio) (6 страниц — по одной на каждый процесс).

### Краткое описание мер по процессам

**Процесс 1: Запись на приём** — HTTPS/TLS 1.3, JWT + RBAC (RECEPTIONIST), pgcrypto для PII, маскирование ФИО в логах, уведомления без PII в теле

**Процесс 1а: Клиентский контракт** — электронная форма с согласием на обработку ПДн, pgcrypto для паспортных данных, электронная подпись PDF, аудит создания/чтения контракта

**Процесс 2: Приём врачом** — JWT с scope=own_patients, RBAC + ABAC (только свои пациенты), pgcrypto для diagnosis/anamnesis (ключ из Vault), Lab API через mTLS + OAuth2 + HMAC

**Процесс 3: Обработка платежа** — Payment Gateway PCI DSS, токенизация карты (без PAN), передача в ERP без PII (только payment_id + amount), полный аудит операций

**Процесс 4: Управление складом** — JWT + RBAC (WAREHOUSE_MANAGER), данные поставщиков зашифрованы (тег CONFIDENTIAL), Column-Level Security (врач не видит цены), аудит операций

**Процесс 5: HR и кадровый учёт** — pgcrypto для паспортных данных/СНИЛС/ИНН (тег PII + HR_SENSITIVE), отдельная схема БД `hr`, ключи в Vault, mTLS к банку без ФИО, отчётность через СБИС/Контур
