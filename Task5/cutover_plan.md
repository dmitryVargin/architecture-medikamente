# Task 5 — Cutover-план миграции

## 1. Выбор стратегии миграции

### Анализ вариантов

| Стратегия | Преимущества | Ограничения для «Медикаменте» |
|---|---|---|
| High-level design | Чёткие границы доменов, хорошая документация | Требует полного переписывания перед запуском; слишком высокий риск для медицины |
| Parallel run | Оба сервиса работают параллельно; можно сверять данные | Двойная нагрузка на персонал; нужна синхронизация данных между системами |
| Branch by abstraction | Постепенный перенос функциональности без даунтайма; можно откатить | Требует абстракционных слоёв; сложнее в реализации |

### Выбранная стратегия: **Branch by Abstraction + Parallel Run (гибрид)**

**Обоснование:**
- Медицинская организация не может позволить себе даунтайм фискальных операций и записи пациентов
- Branch by Abstraction позволяет переносить функциональность поэтапно, не останавливая работу
- Parallel Run применяется на критических этапах (перенос медкарт, ККМ) для верификации данных
- Такой подход минимизирует риск потери данных и обеспечивает возможность отката на каждом этапе

**Принцип Branch by Abstraction для «Медикаменте»:**
```
Вместо прямого обращения сервисов к Excel/1С → вводим Abstraction Layer (Adapter):
  - Adapter читает из старой системы (Excel/1С файловый)
  - Параллельно пишет в новую систему (PostgreSQL/1С серверный)
  - После проверки — переключаем чтение на новую систему
  - После стабилизации — удаляем адаптер и старую систему
```

## 5. Ключевые риски и меры по снижению

| Риск | Вероятность | Влияние | Меры по снижению |
|---|---|---|---|
| Потеря данных при миграции Excel → PostgreSQL | Средняя | Критическое | Полный backup до ETL + верификация 100% записей + параллельный период |
| Отказ ККМ при замене адаптера | Средняя | Критическое (фискальная ответственность) | Замена в нерабочее время + резервный канал OLE как fallback |
| Расхождения данных в период parallel run | Высокая | Высокое | Автоматическая сверка каждый час + алерт при расхождении |
| Сопротивление врачей переходу на МИС | Высокая | Среднее | Демо-сессии, обучение, hotline поддержки |
| Нарушение работы Lab Integration | Низкая | Высокое | Fallback: email + ручная привязка результатов |
| Недоступность серверов в момент cutover | Низкая | Критическое | Cutover только в нерабочее время (выходные), UPS, резервный канал связи |
| Ошибки при шифровании ЭМК (pgcrypto) | Низкая | Критическое | Тестирование на 1000 карт в staging, верификация расшифровки |
| Превышение сроков разработки | Высокая | Среднее | Буфер 20% к каждому этапу, еженедельный статус-митинг |

## 6. Процедура Rollback

### Общие принципы:
- Решение о rollback принимает Технический директор
- Rollback запускается при: потере данных, деградации производительности >50%, недоступности критических функций >30 минут
- Каждый этап имеет собственную процедуру rollback (описана выше)
- Максимальное время rollback: 2 часа

### Процедура экстренного rollback (любой этап):
```
1. Уведомить Технического директора → решение о rollback (5 мин)
2. Переключить трафик на старую систему (DNS / Load Balancer) (10 мин)
3. Восстановить backup если затронуты данные (30-120 мин)
4. Верифицировать работоспособность старой системы (15 мин)
5. Уведомить всех пользователей о возврате (5 мин)
6. Провести post-mortem: причина, что пошло не так (на следующий день)
```
