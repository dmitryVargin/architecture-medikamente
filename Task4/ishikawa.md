# Task 4 — Диаграмма Исикавы: узкие места при миграции

## 1. Постановка проблемы

**Главная проблема (голова рыбы):**
> Снижение производительности и риск потери данных при миграции с монолитной архитектуры на микросервисную

---

## 2. Диаграмма Исикавы

> Визуальная диаграмма Исикавы (fishbone) представлена в файле [ishikawa.drawio](ishikawa.drawio).

**6 категорий причин** (ветви диаграммы): Инфраструктура, Программное обеспечение, Интеграции, Персонал, Процессы, Данные.

---

## 3. Детализация по категориям

### Категория 1: Инфраструктура

| Проблема | Последствие | Приоритет |
|---|---|---|
| Единственный физический сервер (нет HA) | Любая поломка = полная недоступность; миграция требует плановый даунтайм | КРИТИЧЕСКИЙ |
| Нет DR (Disaster Recovery) | Потеря данных при сбое во время миграции | КРИТИЧЕСКИЙ |
| Нет тестовой / staging среды | Невозможно протестировать миграцию без влияния на production | ВЫСОКИЙ |
| Нет системы мониторинга | Сложно выявить деградацию производительности после переключения | ВЫСОКИЙ |
| Отсутствие Kubernetes / оркестрации | Развёртывание микросервисов потребует времени на освоение | СРЕДНИЙ |
| Нет балансировщика нагрузки | При микросервисах — критичный компонент, требует настройки | СРЕДНИЙ |

### Категория 2: Программное обеспечение

| Проблема | Последствие | Приоритет |
|---|---|---|
| 1С в файловом режиме | Прямой доступ к файлам без API; миграция данных крайне затруднена | КРИТИЧЕСКИЙ |
| Excel как основное хранилище | Нет схемы, нет транзакций; трансформация в реляционную БД — трудоёмко | КРИТИЧЕСКИЙ |
| Отсутствие API у текущих систем | Невозможно постепенное отключение без полного переписывания | ВЫСОКИЙ |
| Нет версионирования схемы БД | Нет Flyway/Liquibase; изменения схемы сложно отслеживать | ВЫСОКИЙ |
| Дублирование данных (Excel + 1С) | Расхождения при миграции; непонятно, какой источник «правда» | ВЫСОКИЙ |
| Монолитная бизнес-логика в 1С | Декомпозиция на микросервисы требует глубокого реинжиниринга | СРЕДНИЙ |

### Категория 3: Интеграции

| Проблема | Последствие | Приоритет |
|---|---|---|
| ККМ через OLE/TCP + 1С | Нет современного API; при замене 1С — риск потери фискальной функции | КРИТИЧЕСКИЙ |
| Ручная интеграция с лабораторией | Процесс неавтоматизирован; API-интеграция требует согласования с лабораторией | ВЫСОКИЙ |
| Exchange как транспорт данных (email) | Данные в почте не структурированы; нет возможности автоматической обработки | СРЕДНИЙ |
| Нет единого формата обмена данными | Каждая интеграция «самодельная»; при миграции — риск несовместимости | СРЕДНИЙ |
| OLE-интеграция устаревшая технология | Не поддерживается в Linux/Kubernetes-среде | ВЫСОКИЙ |

### Категория 4: Персонал

| Проблема | Последствие | Приоритет |
|---|---|---|
| Нет опыта с Docker/Kubernetes | Задержки при развёртывании новой инфраструктуры | ВЫСОКИЙ |
| Нет DevOps/Platform Engineer | Настройка CI/CD, K8s, мониторинга займёт значительное время | ВЫСОКИЙ |
| Сопротивление изменениям (Change Management) | Сотрудники продолжат работать со старой системой параллельно | СРЕДНИЙ |
| Зависимость от одного сетевого администратора | Bus factor = 1; риск при болезни / уходе ключевого сотрудника | ВЫСОКИЙ |
| Нет специалиста по ИБ | Новая архитектура будет внедрена без должного аудита безопасности | ВЫСОКИЙ |
| Нет DBA (Database Administrator) | Настройка PostgreSQL, шифрование, репликация — выполнять некому | СРЕДНИЙ |

### Категория 5: Процессы

| Проблема | Последствие | Приоритет |
|---|---|---|
| Нет формального Change Management | Изменения вносятся хаотично; высокий риск регрессий | ВЫСОКИЙ |
| Нет CI/CD pipeline | Ручные деплои ненадёжны; откат занимает много времени | ВЫСОКИЙ |
| Нет процедуры rollback | При неудачной миграции нет плана возврата к рабочему состоянию | КРИТИЧЕСКИЙ |
| Нет SLA/SLO определений | Непонятно, что считать «успешной» миграцией | СРЕДНИЙ |
| Нет тестирования производительности | Нагрузочные тесты отсутствуют; пороговые значения неизвестны | СРЕДНИЙ |
| Нет документации текущей системы | Архитектура AS-IS не документирована; риск пропустить компоненты | ВЫСОКИЙ |

### Категория 6: Данные

| Проблема | Последствие | Приоритет |
|---|---|---|
| Неструктурированные Excel-файлы | Трансформация и очистка займут значительное время | КРИТИЧЕСКИЙ |
| Нет единой схемы данных | Разные форматы в разных файлах; риск потери при конвертации | КРИТИЧЕСКИЙ |
| Дубликаты и ошибки в записях | Нужна дедупликация перед миграцией | ВЫСОКИЙ |
| Нет Data Dictionary | Непонятно, что означает каждое поле в Excel | ВЫСОКИЙ |
| Исторические данные без нормализации | Невозможно напрямую загрузить в реляционную СУБД | ВЫСОКИЙ |
| Отсутствие классификации данных | Непонятно, где в Excel чувствительные данные; риск утечки при миграции | ВЫСОКИЙ |

## 4. Рекомендации и приоритизация

### Срочные меры (до начала миграции)

1. **Развернуть staging-среду** — минимум 1 сервер (виртуальный или облачный) для тестирования
2. **Создать резервную копию всех данных** — полный backup перед любыми изменениями
3. **Перевести 1С на серверный режим** — устраняет зависимость от файловой системы
4. **Документировать AS-IS** — все Excel-файлы, их структуру, зависимости
5. **Разработать процедуру rollback** — для каждого этапа миграции

### Краткосрочные меры (в ходе MVP, 2 месяца)

6. **Нанять DevOps-инженера** — или привлечь подрядчика для K8s/CI-CD
7. **Настроить CI/CD (GitLab CI / GitHub Actions)** — автоматизация деплоев
8. **Установить мониторинг (Victoria Metrics + Grafana)** — базовые метрики до миграции
9. **Начать замену OLE-интеграции ККМ** — разработать адаптер с современным API
10. **Провести Data Quality Assessment** — выявить дубли, ошибки в Excel

### Среднесрочные меры (в ходе финальной миграции, год)

11. **Настроить HashiCorp Vault** — управление секретами и ключами шифрования
12. **Внедрить Istio (mTLS)** — безопасность межсервисного взаимодействия
13. **Нанять ИБ-специалиста** — аудит безопасности новой архитектуры
14. **Обучить сотрудников** — работа с новыми порталами + ИБ-грамотность
15. **Разработать Data Dictionary** — реестр всех данных с классификацией

## 5. Матрица рисков (по приоритету)

| Риск | Вероятность | Влияние | Приоритет | Мера |
|---|---|---|---|---|
| Потеря данных при миграции Excel | Высокая | Критическое | КРИТИЧЕСКИЙ | Полный backup + тестовая миграция |
| Даунтайм фискальной системы (ККМ) | Средняя | Критическое | КРИТИЧЕСКИЙ | Параллельная работа + адаптер |
| Нет rollback-плана | Высокая | Критическое | КРИТИЧЕСКИЙ | Разработать до начала миграции |
| Нет HA → потеря при сбое | Средняя | Высокое | ВЫСОКИЙ | Резервный сервер или облако |
| Сопротивление персонала | Высокая | Среднее | ВЫСОКИЙ | Change Management, обучение |
| Bus factor = 1 (сетевой админ) | Средняя | Высокое | ВЫСОКИЙ | Нанять дополнительного специалиста |
| Данные лаборатории не в API | Низкая | Среднее | СРЕДНИЙ | Параллельный режим во время интеграции |
